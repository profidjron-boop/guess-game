# Дуэль тайминга

Многопользовательская браузерная игра в реальном времени для спортивных и киберспортивных трансляций.  
Игроки нажимают `СЕЙЧАС!` в момент события, система считает отклонение в миллисекундах и автоматически формирует результат раунда, финальную таблицу и глобальный рейтинг.

Проект сделан как готовый продукт: с боевой архитектурой, серверной валидацией критичной логики, устойчивым realtime-потоком и полноценным UX для лобби, раундов и итогов.

## Основные возможности

- Полный мультиплеерный цикл: лобби -> комната -> серия из 5 раундов -> финал -> глобальный лидерборд.
- Realtime-синхронизация комнаты и состояния игры через Supabase Realtime.
- Серверный расчёт времени нажатия и очков через Postgres функции (без доверия к клиенту).
- Защита от повторной отправки нажатий и гонок состояний при старте/обновлении игры.
- Детализированный экран результатов: событие, время нажатия, отклонение, очки, объяснение результата.
- Финальный экран с аналитикой, бейджами, историей раундов и карточкой результата для шаринга.

## Архитектурная идея

Проект разделяет ответственность между UI и серверной логикой:

- **Клиент (Next.js):** интерфейс, анимации, взаимодействие игрока, подписка на realtime-события.
- **База (Supabase/Postgres):** источник истины для комнат, раундов, нажатий и итогов.
- **RPC-функции:** вычисления и валидация критичной игровой логики на сервере.

Такой подход даёт честную механику (невозможно "подрисовать" результат на клиенте), предсказуемую синхронизацию между игроками и стабильное поведение при сетевых сбоях.

## Как работает multiplayer и realtime

1. Хост создаёт комнату, игроки присоединяются по коду/ссылке.
2. Клиенты подписываются на изменения таблиц `rooms`, `participants`, `rounds` через Realtime-канал комнаты.
3. При нажатии `СЕЙЧАС!` клиент вызывает RPC `submit_guess_server(...)`.
4. База валидирует действие (раунд активен, игрок в комнате, ещё не нажимал) и считает `press_time_ms` + `delta_ms`.
5. После завершения раунда применяется `apply_round_results(...)` (очки, серии, победитель раунда).
6. В финале вызывается `finalize_game(...)`, данные записываются в `leaderboard`.

Дополнительно в UI реализованы состояния `Подключено / Переподключаемся… / Соединение потеряно`, чтобы пользователь всегда понимал статус соединения.

## Настройка Supabase

1. Создайте новый проект в Supabase.
2. Откройте SQL Editor и выполните файл `supabase/schema.sql`.
3. Убедитесь, что Realtime включён для таблиц:
   - `rooms`
   - `participants`
   - `rounds`
   - `guesses`
   - `leaderboard`

Примечание: проект рассчитан на гостевых игроков (без обязательной авторизации).  
RLS и policies уже входят в `supabase/schema.sql`.

## Переменные окружения

Скопируйте шаблон и заполните значения из вашего Supabase-проекта:

```bash
cp .env.example .env.local
```

Нужны переменные:

- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`

## Применение SQL schema

Вариант через Dashboard:

1. Откройте `Supabase -> SQL Editor`.
2. Скопируйте содержимое `supabase/schema.sql`.
3. Выполните скрипт целиком.

Что создаётся:

- Таблицы: `rooms`, `participants`, `round_templates`, `rounds`, `guesses`, `leaderboard`.
- Индексы и связи.
- Функции: `compute_base_points`, `submit_guess_server`, `apply_round_results`, `finalize_game`.
- RLS и guest-friendly policies.
- Начальные шаблоны раундов.

## Локальный запуск

```bash
npm install
npm run dev
```

Приложение будет доступно на [http://localhost:3000](http://localhost:3000).

Для проверки мультиплеера:

1. Откройте две вкладки/два браузера.
2. В первой вкладке создайте комнату.
3. Во второй присоединитесь по коду.
4. Запустите игру и проверьте синхронность раундов, результатов и финальной таблицы.

## Engineering decisions

- **Серверный расчёт критичных значений:** `delta` и очки считаются в Postgres-функциях, чтобы исключить клиентские манипуляции.
- **Оптимистичный, но контролируемый UI:** пользователь получает мгновенный отклик, но финальные данные всегда приходят из базы.
- **Практичная устойчивость realtime:** дебаунс обновлений, защита от гонок и явные connection states для UX без "сломанных" экранов.
- **Mobile-first интерфейс:** ключевые действия доступны и читаемы на маленьких экранах.
