# Дуэль тайминга

> **Главное для смысла:** игрок смотрит **реальную трансляцию** на другом экране и нажимает `СЕЙЧАС!`, когда **видит** событие (гол, килл и т.д.). **Эталон времени события** для очков в демо задаёт **хост**, нажав кнопку в момент, когда событие произошло на его эфире (аналог внешнего таймкода; позже можно заменить на API стрима). Очки — по разнице между эталоном и моментом нажатия. Отдельного таймера «окна раунда» в механике нет — ориентир только эфир.

**Second-screen companion** для болельщика: трансляцию вы смотрите **не в этом окне**, а рядом; здесь — каталог матчей, выбор стороны, комната с друзьями и кнопка в момент события на эфире (гол, килл и т.д.). Сервер фиксирует отклонение в миллисекундах, считает очки и ведёт рейтинг.

Игровой **раунд** — смена цели и эталона хоста по ходу матча; ориентир — **картинка и звук трансляции** и подпись события в интерфейсе.

Стек: Next.js, Supabase (Postgres + Realtime + RPC), валидация критичной логики на сервере, UX для каталога матчей, комнаты и серии раундов.

## Продуктовый поток

1. **Каталог матчей** — лига, статус, пары команд.
2. **Карточка матча** — контекст трансляции, выбор стороны (домашние / гости).
3. **Комната** — код/ссылка, участники с выбранной командой; в данных комнаты сохраняются привязка к матчу (`match_slug`, лига, тип события).
4. **Игра** — серия раундов; на экране отображаются матч, лига, сторона и целевое событие раунда.
5. **Финал и лидерборд** — итоги сессии и глобальная таблица.

В демо каталог матчей может подставляться из **моков** в коде; при наличии данных в Supabase таблица `matches` и шаблоны `match_event_templates` задают тот же контекст на стороне БД.

## Основные возможности

- Полный цикл: каталог матчей → комната → серия раундов → финал → глобальный лидерборд.
- Realtime-синхронизация комнаты и состояния игры через Supabase Realtime.
- Серверный расчёт времени нажатия и очков через Postgres (клиент не является источником истины для результата).
- Защита от повторных нажатий и гонок при старте/обновлении игры.
- Экраны результатов: событие, время нажатия, отклонение, очки.
- Финальный экран с аналитикой, бейджами и историей раундов.

## Архитектурная идея

- **Клиент (Next.js):** интерфейс, анимации, ввод игрока, подписка на realtime.
- **База (Supabase/Postgres):** источник истины для комнат, раундов, нажатий, итогов и (при использовании) каталога матчей.
- **RPC-функции:** валидация и расчёт критичной игровой логики на сервере.

Так достигается согласованность между игроками и предсказуемое поведение при сетевых сбоях.

## Как работает multiplayer и realtime

1. Игрок выбирает матч и сторону, хост создаёт комнату (контекст матча уезжает в поля `rooms`).
2. Остальные присоединяются по коду/ссылке; у каждого участника фиксируется выбранная команда.
3. Клиенты подписываются на изменения `rooms`, `participants`, `rounds` в канале комнаты.
4. По `СЕЙЧАС!` вызывается RPC `submit_guess_server(...)` (фиксируется `press_time_ms` относительно старта раунда; `delta_ms` заполняется после эталона).
5. Хост вызывает `mark_round_event(...)` в момент события на трансляции — задаётся `event_time_ms`, пересчитываются дельты, затем `apply_round_results(...)`.
6. В конце игры — `finalize_game(...)` и запись в `leaderboard`.

В UI отображаются состояния соединения (`Подключено` / переподключение / потеря), чтобы не оставлять пользователя без обратной связи.

## Настройка Supabase

1. Создайте проект в Supabase.
2. В SQL Editor выполните `supabase/schema.sql`.
3. Включите Realtime для нужных таблиц:
   - **Каталог матчей (если список матчей ведёте в БД, не только моками):** `matches`, `match_event_templates`.
   - **Игровой цикл (обязательно для мультиплеера):** `rooms`, `participants`, `rounds`, `guesses`, `leaderboard`.

   Первые две таблицы — отдельный **продуктовый слой «матч»**; без них в коде всё равно есть маршруты и мок-каталог, но в схеме они присутствуют и связывают комнату с матчем (`match_slug`, см. `supabase/schema.sql`).

Примечание: сценарий с гостями без обязательной авторизации; RLS и policies входят в `supabase/schema.sql`.

## Применение SQL schema

1. `Supabase → SQL Editor`.
2. Вставьте содержимое `supabase/schema.sql` и выполните.

**Что появляется в базе**

| Слой                 | Таблицы                                                     | Зачем                                                                                                             |
| -------------------- | ----------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------- |
| **Матчи**            | `matches`, `match_event_templates`                          | Каталог матчей и шаблоны событий; связь с UI `/`, `/match/[slug]`.                                                |
| **Раунды (шаблоны)** | `round_templates`                                           | Базовые 5 раундов по умолчанию.                                                                                   |
| **Игра**             | `rooms`, `participants`, `rounds`, `guesses`, `leaderboard` | Комнаты, игроки, раунды, нажатия, итоги; в `rooms`/`rounds` — поля контекста матча (`match_slug`, лига, событие). |

Полный перечень одной строкой: `matches` → `match_event_templates` → `round_templates` → `rooms` → `participants` → `rounds` → `guesses` → `leaderboard`.

Также в скрипте: индексы; RPC `compute_base_points`, `submit_guess_server`, `mark_round_event`, `apply_round_results`, `finalize_game`; RLS для гостевого режима; начальные строки в `round_templates`.

## Engineering decisions

- **Серверный расчёт:** `delta` и очки в Postgres, без доверия к подделке на клиенте.
- **Оптимистичный UI с подтверждением из БД:** мгновенный отклик, итог с сервера.
- **Realtime:** дебаунс обновлений, защита от гонок, явные состояния соединения.
- **Mobile-first:** ключевые действия доступны на маленьких экранах.
