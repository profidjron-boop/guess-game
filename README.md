# Дуэль тайминга

**Second-screen companion** для болельщика во время конкретной спортивной или киберспортивной трансляции: выбираешь матч и сторону, заходишь в комнату с друзьями и нажимаешь `СЕЙЧАС!`, когда на экране происходит заранее названное событие (гол, килл и т.д.). Сервер фиксирует отклонение в миллисекундах, считает очки и ведёт рейтинг.

Игровой **раунд** в приложении — это техническое окно синхронизации между игроками (длительность раунда в UI), а не «замена» реального времени матча. Ориентир для игрока — **трансляция и описание события**; внутренний таймер лишь ограничивает фазу, в которой принимаются нажатия.

Стек: Next.js, Supabase (Postgres + Realtime + RPC), серверная валидация критичной логики, UX для каталога матчей, лобби комнаты и серии раундов.

## Продуктовый поток

1. **Каталог матчей** — лига, статус, пары команд.
2. **Карточка матча** — контекст трансляции, выбор стороны (домашние / гости).
3. **Комната** — код/ссылка, участники с выбранной командой; в данных комнаты сохраняются привязка к матчу (`match_slug`, лига, тип события).
4. **Игра** — серия раундов; на экране отображаются матч, лига, сторона и целевое событие раунда.
5. **Финал и лидерборд** — итоги сессии и глобальная таблица.

В демо каталог матчей может подставляться из **моков** в коде; при наличии данных в Supabase таблица `matches` и шаблоны `match_event_templates` задают тот же контекст на стороне БД.

## Основные возможности

- Полный цикл: каталог матчей → комната → серия раундов → финал → глобальный лидерборд.
- Realtime-синхронизация комнаты и состояния игры через Supabase Realtime.
- Серверный расчёт времени нажатия и очков через Postgres (клиент не является источником истины для результата).
- Защита от повторных нажатий и гонок при старте/обновлении игры.
- Экраны результатов: событие, время нажатия, отклонение, очки.
- Финальный экран с аналитикой, бейджами и историей раундов.

## Архитектурная идея

- **Клиент (Next.js):** интерфейс, анимации, ввод игрока, подписка на realtime.
- **База (Supabase/Postgres):** источник истины для комнат, раундов, нажатий, итогов и (при использовании) каталога матчей.
- **RPC-функции:** валидация и расчёт критичной игровой логики на сервере.

Так достигается согласованность между игроками и предсказуемое поведение при сетевых сбоях.

## Как работает multiplayer и realtime

1. Игрок выбирает матч и сторону, хост создаёт комнату (контекст матча уезжает в поля `rooms`).
2. Остальные присоединяются по коду/ссылке; у каждого участника фиксируется выбранная команда.
3. Клиенты подписываются на изменения `rooms`, `participants`, `rounds` в канале комнаты.
4. По `СЕЙЧАС!` вызывается RPC `submit_guess_server(...)`.
5. База проверяет: раунд активен, игрок в комнате, повторного нажатия не было; считаются `press_time_ms` и `delta_ms` относительно серверного окна раунда.
6. После раунда — `apply_round_results(...)`, в конце игры — `finalize_game(...)` и запись в `leaderboard`.

В UI отображаются состояния соединения (`Подключено` / переподключение / потеря), чтобы не оставлять пользователя без обратной связи.

## Настройка Supabase

1. Создайте проект в Supabase.
2. В SQL Editor выполните `supabase/schema.sql`.
3. Убедитесь, что Realtime включён для таблиц:
   - `rooms`
   - `participants`
   - `rounds`
   - `guesses`
   - `leaderboard`

Примечание: сценарий с гостями без обязательной авторизации; RLS и policies входят в `supabase/schema.sql`.

## Переменные окружения

```bash
cp .env.example .env.local
```

- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`

## Применение SQL schema

1. `Supabase → SQL Editor`.
2. Вставьте содержимое `supabase/schema.sql` и выполните.

Создаётся в том числе:

- **Ядро игры:** `rooms` (в т.ч. поля контекста матча), `participants` (в т.ч. выбранная сторона), `round_templates`, `rounds` (в т.ч. `event_type`, `event_label`, привязка к матчу), `guesses`, `leaderboard`.
- **Каталог матчей (second-screen):** `matches`, `match_event_templates`.
- Индексы, RPC: `compute_base_points`, `submit_guess_server`, `apply_round_results`, `finalize_game`.
- RLS и политики для гостевого режима.
- Начальные шаблоны раундов.

## Engineering decisions

- **Серверный расчёт:** `delta` и очки в Postgres, без доверия к подделке на клиенте.
- **Оптимистичный UI с подтверждением из БД:** мгновенный отклик, итог с сервера.
- **Realtime:** дебаунс обновлений, защита от гонок, явные состояния соединения.
- **Mobile-first:** ключевые действия доступны на маленьких экранах.
